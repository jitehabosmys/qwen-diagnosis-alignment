# 恒源云 50 元预算微调实践执行计划

## 1. 项目目标

在恒源云单卡 `4090 24GB` 环境下，用约 `50 元` 预算完成一套尽量完整、可复现、可解释的大模型实践流程：

- 选择合适的开源基座模型
- 准备一套自定义中文训练数据
- 完成 `SFT`
- 完成一轮轻量级偏好优化，优先选 `DPO`
- 用 `vLLM` 做推理验证
- 如预算有余，补一个极小规模的规则奖励实验，作为 `GRPO` 入门

本项目不追求“大而全”，而追求：

- 以最低可行预算跑通完整链路
- 对每一步为什么这样做有清晰理解
- 最后能产出一套自己能讲明白的实验记录

## 2. 现实约束

### 2.1 已知条件

- 预算：`50 元`
- GPU 价格：约 `1.5 元/小时`
- 可用 GPU 总时长：约 `33.3 小时`
- 当前目标：练习 `LoRA / QLoRA + SFT + DPO + 部署`
- 可选扩展：少量 `GRPO` 验证

### 2.2 结论

本次项目建议主线为：

`Qwen3-4B-Instruct + QLoRA + 自定义中文结构化任务 + SFT + DPO + vLLM`

不建议作为主线的内容：

- `8B` 作为第一次完整实践的主模型
- `PPO`
- 大规模公开数据混训
- 第一次就训 reward model
- 第一次就做复杂多任务混合训练

## 3. 推荐任务方向

### 3.1 任务主题

建议做：

**中文训练/部署报错诊断助手**

输入示例：

- 训练命令
- 环境信息
- 报错日志
- 用户自然语言提问

输出示例：

- 问题分类
- 严重程度
- 一句话总结
- 最可能根因
- 缺失信息
- 下一步排查建议

### 3.2 为什么选这个方向

- 与当前学习主题高度相关：训练、微调、推理、部署
- 中文场景更适合 `Qwen`
- 自己可以写出高质量种子数据
- `SFT` 容易看到收益
- `DPO` 数据容易构造
- 能设计规则打分，给后续 `GRPO` 预留接口
- 最终部署效果直观，适合用 `vLLM` 演示

### 3.3 输出格式建议

建议输出固定 JSON：

```json
{
  "category": "env|dependency|data|training|inference|deployment",
  "severity": "low|medium|high",
  "summary": "一句话结论",
  "root_cause": "最可能的根因",
  "missing_info": ["还缺哪些关键信息"],
  "next_steps": ["可执行的排查建议"]
}
```

这个格式很适合：

- `SFT` 学格式和风格
- `DPO` 学偏好
- 后续做规则校验
- 自动评测

## 4. 模型与工具选型

### 4.1 基座模型

首选：

- `Qwen3-4B-Instruct`

原因：

- 对中文更友好
- 相比 `8B` 更适合单卡低预算反复试验
- 更容易把 GPU 时间留给 `DPO`、评测和部署

### 4.2 训练方式

首选：

- `QLoRA`

原因：

- 更省显存
- 更适合 `4090 24GB`
- 更适合预算有限的多轮试验

### 4.3 工具链

建议主工具：

- `LLaMA-Factory`：快速跑通 `SFT + DPO`
- `vLLM`：推理与部署验证

建议后续学习工具：

- `TRL`：用于理解 `DPO / GRPO` 训练器机制

### 4.4 为什么先不用更底层方案

第一次项目最重要的是：

- 先跑通
- 再理解
- 再重写

`LLaMA-Factory` 不会降低项目的“专业度”。真正体现专业度的是：

- 数据定义是否清楚
- 配置是否可复现
- 评测是否客观
- 实验是否有对照
- 是否知道哪里有效、哪里无效

## 5. 数据方案

### 5.1 数据类型

本项目至少准备三份数据：

1. `SFT` 训练集
2. `DPO` 偏好数据
3. `Eval` 验证集

### 5.2 目标规模

建议首轮规模：

- `SFT`: `800-1500` 条
- `DPO`: `500-1000` 对
- `Eval`: `100-200` 条

这个规模足够小，适合低预算快速迭代；同时又足够大，能让 `SFT` 和 `DPO` 有可见效果。

### 5.3 数据来源

#### A. 手工高质量种子集

先自己写 `150-300` 条。

推荐覆盖：

- `pip` / `conda` 安装失败
- `CUDA out of memory`
- `bitsandbytes` 相关报错
- `transformers` / `peft` / `trl` 版本冲突
- tokenizer / chat template 配置错误
- 数据格式不合法
- `jsonl` 字段缺失
- `vLLM` 启动失败
- 推理时报参数不兼容
- 训练 loss 异常
- 显卡、驱动、CUDA 版本不匹配

#### B. 模型扩写

用基座模型将种子数据扩写到 `800-1500` 条。

扩写方式：

- 改写用户描述风格
- 改写命令行参数
- 改写日志细节
- 改写环境组合
- 改写信息完整度
- 改写报错阶段

#### C. DPO 对构造

推荐先写好“好答案”，再系统构造“坏答案”。

坏答案模板：

- JSON 非法
- 分类错误
- 根因过度自信
- 忽略缺失信息
- 下一步建议空泛
- 凭空引入日志中不存在的信息
- 没有遵守 schema

### 5.4 数据存储压力

这类项目的数据体量通常很小：

- 自定义数据一般只有几十 MB 到几百 MB
- 中间产物加起来通常也是 GB 级

结论：

**本地和服务器都装得下，磁盘不是主要问题。**

主要瓶颈是：

- GPU 小时
- 数据质量
- 管道是否稳定

## 6. 评测方案

### 6.1 核心评测目标

不要只看训练 loss。

更重要的是：

- 输出是不是合法 JSON
- 字段是否齐全
- 分类是否合理
- 根因是否贴近输入
- 是否会承认信息不足
- 排查步骤是否可执行

### 6.2 第一版自动指标

建议先实现以下规则评测：

1. `json_parse_success`
2. `required_fields_present`
3. `enum_fields_valid`
4. `missing_info_nonempty`
5. `next_steps_nonempty`
6. `schema_extraneous_fields`

### 6.3 第一版人工评测

建议从 `Eval` 集中抽 `30-50` 条，做人工打分：

- 是否理解问题
- 是否抓住最可能根因
- 是否有明显幻觉
- 建议是否可执行

### 6.4 对比对象

至少做三组对比：

1. `Base`
2. `SFT`
3. `SFT + DPO`

如果做了 `GRPO`，再加：

4. `SFT + DPO + GRPO`

## 7. 预算与 GPU 时间分配

### 7.1 预算上限

- 总预算：`50 元`
- GPU 单价：`1.5 元/小时`
- 理论总时长：约 `33.3 小时`

### 7.2 推荐拆分

#### 阶段 A：冒烟测试

- 预算：`3-4.5 元`
- 时间：`2-3 小时`

内容：

- 模型加载
- tokenizer 检查
- 少量样本训练
- adapter 保存和加载

#### 阶段 B：正式 SFT

- 预算：`12-18 元`
- 时间：`8-12 小时`

内容：

- 首轮完整 `SFT`
- 保存 checkpoint
- 简单评测

#### 阶段 C：正式 DPO

- 预算：`9-15 元`
- 时间：`6-10 小时`

内容：

- 小规模偏好训练
- 对比 base / SFT / DPO

#### 阶段 D：vLLM 推理验证

- 预算：`3-6 元`
- 时间：`2-4 小时`

内容：

- 加载基座或 adapter
- 接口形式推理
- 简单对比测试

#### 阶段 E：返工与缓冲

- 预算：`3-6 元`
- 时间：`2-4 小时`

内容：

- 修配置
- 补跑
- 重训一小轮

#### 阶段 F：可选 GRPO

- 预算：`0-6 元`
- 时间：`0-4 小时`

内容：

- 仅在前面都顺利时尝试
- 仅做小规模规则奖励实验

### 7.3 总体原则

GPU 时间优先留给：

- 正式 `SFT`
- 正式 `DPO`
- 最终推理对比

GPU 时间不要浪费在：

- 装环境
- 下载依赖
- 下载模型
- 修数据格式
- 改 YAML
- 排查字段名错误

## 8. 本地、无卡实例、GPU 实例的分工

### 8.1 本地完成的工作

建议在本地完成：

- 任务设计
- schema 设计
- 数据样本编写
- 数据清洗脚本
- 评测脚本
- 训练配置草稿

### 8.2 无卡实例完成的工作

建议在无卡或不开机阶段完成：

- 目录规划
- 环境准备方案
- 依赖列表整理
- 下载脚本准备
- 数据格式检查
- 训练配置审查

### 8.3 开 GPU 之后才做的工作

- 模型推理 sanity check
- 小样本训练冒烟
- 正式 `SFT`
- 正式 `DPO`
- `vLLM` 推理测试
- 可选 `GRPO`

### 8.4 判断标准

只有在满足下列条件后再开 GPU：

- 数据格式已确定
- 配置文件已写好
- 训练命令已确定
- 输出目录已规划
- 评测脚本已准备
- 至少完成一次 CPU 侧 dry-run

## 9. 分阶段执行计划

### 阶段 0：项目初始化

目标：

- 建立目录结构
- 写清任务定义
- 固定输出 schema

产物：

- `README` 或任务说明
- `docs/` 中的项目计划
- 数据目录
- 实验目录

建议目录：

```text
project/
  docs/
  data/
    raw/
    sft/
    dpo/
    eval/
  scripts/
  configs/
  outputs/
  reports/
```

### 阶段 1：准备种子数据

目标：

- 手写 `150-300` 条高质量样本

要求：

- 题型覆盖多样
- 日志尽量真实
- 输出格式严格统一
- 每条样本都能解释为什么这样标注

建议先做小批量评审：

- 先写 `30-50` 条
- 检查 schema 是否合理
- 再扩大到 `150-300` 条

### 阶段 2：扩写 SFT 数据

目标：

- 将数据扩到 `800-1500` 条

注意：

- 不要盲目扩写
- 每次扩写后要抽样检查
- 去掉明显模板化、低质量、重复样本

### 阶段 3：构造 DPO 数据

目标：

- 做出 `500-1000` 对偏好样本

方法：

- 从 `SFT` 数据衍生
- 对每条好答案制造一个或多个坏答案
- 保证好坏差异清晰，不要“都差不多”

### 阶段 4：准备验证集

目标：

- 留出 `100-200` 条独立验证数据

要求：

- 不参与训练
- 覆盖常见和少见错误
- 同时包含信息充分和信息不足两类问题

### 阶段 5：无卡 dry-run

目标：

- 在不开 GPU 的前提下排除低级错误

检查项：

- 数据字段名是否一致
- prompt 模板是否正确
- tokenizer 是否能正常处理
- 训练配置是否能被框架识别
- 输出目录和日志目录是否正确

### 阶段 6：GPU 冒烟测试

目标：

- 用最小成本确认全链路可运行

建议：

- 只取 `50-100` 条样本
- 只跑很短一轮
- 验证 checkpoint、adapter、推理加载是否都正常

成功标准：

- 能正常开始训练
- 能正常保存
- 能正常加载
- 微调后回答风格有轻微变化

### 阶段 7：正式 SFT

目标：

- 让模型稳定学会输出指定 JSON 结构

训练重点：

- 格式正确性
- 任务理解
- 输出收敛
- 减少胡言乱语

结束后要做的事：

- 保存最好 checkpoint 或 adapter
- 在 `Eval` 集上跑自动评测
- 抽样做人工评审

### 阶段 8：正式 DPO

目标：

- 提升输出稳定性和偏好质量

训练重点：

- 承认信息不足
- 减少幻觉
- 更倾向可执行建议
- 更稳定地遵守 schema

结束后要做的事：

- 跑 base / SFT / DPO 三组对比
- 记录每组在自动指标上的表现
- 抽样做人工比较

### 阶段 9：vLLM 推理验证

目标：

- 验证模型是否具备可部署性

重点观察：

- 延迟
- 输出稳定性
- 长日志输入时的表现
- 多轮测试时的格式稳定度

### 阶段 10：可选 GRPO 实验

只有在前述阶段顺利完成，且 GPU 预算仍充足时再做。

建议只针对“强约束格式”设计规则奖励，例如：

- JSON 是否可解析
- 是否含必填字段
- 枚举值是否合法
- 不允许出现 schema 外字段
- `missing_info` 至少有一项
- `next_steps` 至少有两项

本阶段目标不是“让模型变更聪明”，而是体验：

- 规则奖励怎么设计
- 可验证 RL 任务是什么感觉
- `GRPO` 在简单任务上的作用边界

## 10. 风险与应对

### 风险 1：GPU 时间被环境问题吃掉

应对：

- 训练前尽量完成所有 dry-run
- 训练和推理环境拆开
- 提前下载依赖和模型

### 风险 2：数据格式不统一

应对：

- 先做小样本
- 写数据校验脚本
- 固定 schema，不轻易中途变更

### 风险 3：SFT 有效果，DPO 效果不明显

应对：

- 优先检查偏好对质量
- 确保 chosen / rejected 差异足够明显
- 先补数据，再考虑延长训练

### 风险 4：任务太难，评测不稳定

应对：

- 保持结构化输出
- 用规则指标约束任务边界
- 不把任务做成万能聊天助手

### 风险 5：GRPO 成本超支

应对：

- 明确 `GRPO` 是 bonus，不是主线
- 主线以 `SFT + DPO + vLLM` 为项目完成标准

## 11. 项目完成标准

满足以下条件即可视为本轮项目成功：

1. 选定一个清晰、可描述的任务
2. 做出一套自己的中文 `SFT` 数据
3. 做出一套自己的 `DPO` 数据
4. 跑通 `Qwen3-4B-Instruct + QLoRA` 的 `SFT`
5. 跑通至少一轮 `DPO`
6. 有一份 `Base vs SFT vs DPO` 的评测结果
7. 能用 `vLLM` 完成一次推理验证
8. 有一份简短实验复盘

- 若预算充足，再尝试极小规模 `GRPO`

## 14. 当前拍板结论

本轮项目采用如下主线：

`Qwen3-4B-Instruct + QLoRA + 中文训练/部署报错诊断任务 + SFT + DPO + vLLM`

其中：

- `SFT` 是主训练阶段
- `DPO` 是主偏好优化阶段
- `vLLM` 是项目闭环的一部分
- `GRPO` 是预算允许时的可选扩展

这条路线在当前预算下最均衡，既能学到关键概念，也最有机会完整跑通。
