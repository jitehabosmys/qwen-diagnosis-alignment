# 2026-04-30 Qwen2.5-0.5B vs 3B Pairwise Judge 结果

这份文档记录的是 `2026-04-30` 基于 `LLM-as-a-judge` 对两组默认 prompt LoRA 输出做的 pairwise 比较结果。

本轮比较对象：

- `qwen25_05b_default_prompt_lora`
- `qwen25_3b_default_prompt_lora`

这轮比较的目标非常明确：

- 判断 `3B` 是否已经显著优于 `0.5B`
- 判断后续主实验路线是否应切换到 `3B`
- 判断后续 DPO 候选采集是否应优先围绕 `3B`

## 1. 比较前提

这轮比较是在两者都已经完成 `SFT` 并经过自动规则评测之后进行的。

因此这里比较的重点不再是：

- 能不能输出 JSON
- schema 是否正确

而是更高一层的内容质量：

- 是否贴近证据
- 根因是否抓得更准
- 排查建议是否更可执行
- 缺失信息提得是否更有用
- 整体是否更像一个合格的工程诊断助手

## 2. 总体结果

本轮 `48` 对样本的 pairwise judge 汇总结果如下：

- `qwen25_05b_default_prompt_lora` 胜：`9`
- `qwen25_3b_default_prompt_lora` 胜：`38`
- `tie`: `1`

去掉 `tie` 之后的胜率：

- `qwen25_05b_default_prompt_lora = 19.15%`
- `qwen25_3b_default_prompt_lora = 80.85%`

这不是轻微优势，而是**非常明显的优势**。

## 3. judge 来源分布

结果中：

- `openai_pairwise_judge = 43`
- `gate_schema_validity = 5`

说明：

- 大多数样本已经进入了真正的内容比较
- 还有一部分样本甚至在 schema gate 阶段就已经分出胜负

这也说明 `3B` 的优势不只是“内容略好”，而是连进入内容比较之前的结构稳定性也更强。

## 4. 五个维度的结果

### 4.1 `evidence_groundedness`

- `0.5B lora = 9`
- `3B lora = 33`
- `tie = 6`

说明：

- `3B` 更常被 judge 认为严格依据输入问题、环境、命令和日志作答
- 更少凭空发挥

### 4.2 `root_cause_quality`

- `0.5B lora = 9`
- `3B lora = 30`
- `tie = 9`

说明：

- `3B` 在抓主要矛盾、给出更有解释力根因上明显更强

### 4.3 `actionability`

- `0.5B lora = 9`
- `3B lora = 34`
- `tie = 5`

说明：

- `3B` 的排查建议更常被 judge 认为具体、保守、可执行

### 4.4 `missing_info_quality`

- `0.5B lora = 8`
- `3B lora = 33`
- `tie = 7`

说明：

- `3B` 提出的缺失信息更有助于下一步定位
- `0.5B` 更容易提出泛化、机械式补问

### 4.5 `overall_engineering_quality`

- `0.5B lora = 8`
- `3B lora = 35`
- `tie = 5`

说明：

- judge 明显更倾向认为 `3B` 的整体回答更像一个合格的训练/部署问题诊断助手

这是当前最值得重视的综合指标。

## 5. 如何解读这轮结果

这轮结果说明的不是：

- `3B` 只是在某个细项上略优

而是：

- `3B` 在五个核心内容质量维度上都表现出明显优势

因此它支持一个非常明确的判断：

**`Qwen2.5-3B-Instruct + LoRA` 已经不是“可能比 0.5B 更好”，而是“在当前任务上明显优于 0.5B LoRA”。**

## 6. 与前面自动规则评测的关系

前面的自动规则评测已经说明：

- `3B base` 本身就比 `0.5B base` 更容易输出合法 JSON
- `3B lora` 的 schema 通过率也很高

而这轮 pairwise judge 进一步说明：

- `3B` 的优势不只体现在格式层
- 在内容质量层也全面领先

把两层结果合在一起，当前最稳妥的结论是：

1. `0.5B` 更适合作为流程验证模型
2. `3B` 已经具备成为主实验模型的条件

## 7. DPO 候选价值

这轮 pairwise judge 还给出了：

- `usable_for_dpo_count = 47`
- `high_confidence_for_dpo_count = 28`

这说明：

- 已经有一批质量不错的偏好对可供后续 DPO 使用
- 而且这些偏好对更值得围绕 `3B` 路线继续积累

换句话说：

**如果后续要进入正式 DPO，优先围绕 `3B` 的输出收集偏好对，比继续围绕 `0.5B` 更合理。**

## 8. 分数层面的额外观察

这轮升级后的 judge 还输出了维度分数与整体分数。

从当前结果看：

- `qwen25_05b_default_prompt_lora` 平均总分约为 `2.6512 / 5`
- `qwen25_3b_default_prompt_lora` 平均总分约为 `2.7442 / 5`
- 总体平均分差约为 `+0.093`

这条观察非常重要，因为它说明：

1. `3B` 相比 `0.5B` 的确明显更强
2. 但两者的绝对分数都还停留在 `2.x / 5`
3. 也就是说，当前系统虽然已经从 `0.5B` 升级到更强底座，但整体内容质量仍未达到真正令人满意的水平

这意味着：

- `3B` 应该成为当前主实验模型
- 但不能把“3B 赢了 0.5B”误解成“问题已经解决”

更准确的说法是：

**scale up 已经带来了可见收益，但收益还不足以让模型达到高质量终态。**

这也正是后续需要继续观察的重点：

- `4B` 是否还能带来有意义的平均分提升
- 还是说此时应该把重心转向更高质量数据、judge 偏好对和后续 `DPO`

## 9. 当前路线判断

这轮结果已经足以推动路线收束：

- `0.5B`
  - 保留为流程验证模型
  - 用于低成本验证训练、评测、judge、DPO 采集链路

- `3B`
  - 作为当前主实验模型
  - 优先承接后续推理评测、judge 和潜在 DPO 主线

## 10. 下一步建议

基于这轮结果，后续建议如下：

1. 将 `3B` 确认为当前主实验路线
2. 后续 judge 与偏好对采集优先围绕 `3B`
3. 如果资源允许，再考虑：
   - 是否继续做 `4B SFT`
   - 是否进入正式 `DPO`

## 11. 一句话结论

这轮 `48` 对 pairwise judge 表明：

**`Qwen2.5-3B-Instruct + LoRA` 以 `38:9:1` 显著优于 `Qwen2.5-0.5B-Instruct + LoRA`，并在五个内容质量维度上全面占优，因此后续主实验路线应切换到 `3B`；但两者平均分仍停留在 `2.x / 5`，说明“更大显著更好”不等于“已经足够好”，后续仍需继续观察更大模型的边际收益与 alignment 路线的价值。**
