# 2026-05-01 HF vs vLLM 小样本观察

这份文档记录的是在 `Qwen3-4B strict lora` 路线上，对 `HF` 推理链路和 `vLLM` 推理链路做的最小样本对比观察。

这轮实验不是正式模型评测，而是一个基础设施验证实验，目标是回答：

1. `vLLM` 是否确实更快
2. `vLLM` 的输出行为是否与当前 `HF` 主评测链足够一致
3. 当前阶段是否适合直接把 `vLLM` 接入主评测流程

## 1. 比较对象

同一模型：

- `Qwen3-4B-Instruct-2507 + LoRA`

同一路线：

- `strict_json_prompt`

同一批样本：

- `eval_001 ~ eval_005`

比较 backend：

- `HF`
- `vLLM`

## 2. 速度观察

当前 `5` 条小样本的平均耗时：

- `HF` 平均约 `9.81s`
- `vLLM` 平均约 `5.16s`

这说明：

- 在当前小样本上，`vLLM` 的速度提升是明确的
- 大致已经接近“快一倍”的量级

因此，从吞吐潜力角度看，`vLLM` 值得继续研究。

## 3. 输出一致性观察

虽然速度明显更快，但小样本内容对比显示：

- `vLLM` 的输出行为和当前 `HF` 主链路并不完全一致

这种不一致不是简单的措辞差异，而是已经涉及：

- 字段缺失
- 字段类型变化
- `category` 判定差异
- 中英文风格变化

## 4. 代表性现象

### 4.1 `eval_001`

HF 版本：

- 中文输出
- `root_cause` 为字符串
- 当前 schema 能通过

vLLM 版本：

- 英文输出
- `root_cause` 变成数组
- 很可能在 strict schema 下失败

这说明：

- 两边不是简单的语言风格差异
- 协议实现方式也发生了变化

### 4.2 `eval_002`

HF 版本：

- `category = dependency`

vLLM 版本：

- `category = deployment`

这说明：

- 后端切换后，不只是格式，连语义判断也可能出现偏差

### 4.3 `eval_003`

HF 版本：

- 包含 `summary`
- 包含 `root_cause`

vLLM 版本：

- 缺少 `summary`
- 缺少 `root_cause`

这会直接影响 strict schema 通过率。

### 4.4 `eval_004`

HF 版本：

- `category = dependency`

vLLM 版本：

- `category = training`
- `root_cause` 仍写成数组

这说明差异不是偶发现象，而是已经出现在多个样本里。

### 4.5 `eval_005`

HF 版本：

- 结构完整

vLLM 版本：

- 缺少 `summary`
- 只保留 `root_cause / missing_info / next_steps`

再次说明：

- `vLLM` 输出并不是简单的“更快但等价”

## 5. 当前最稳的判断

这轮实验可以比较明确地得出两个结论：

### 5.1 速度结论

**`vLLM` 在当前 `4B strict lora` 路线下表现出明显的速度优势。**

这点已经成立。

### 5.2 一致性结论

**当前接入方式下，`vLLM` 的输出行为与 `HF` 主评测链存在不可忽略的偏差。**

而且这种偏差不是轻微风格变化，而是会影响：

- schema
- 字段类型
- 字段完整性
- 语义分类

## 6. 为什么不能立刻替换主评测链路

当前主评测链路的价值在于：

- 训练结果
- 自动规则评测
- pairwise judge

这些结论都是在 `HF` 路线上逐步建立起来的。

如果现在直接切到 `vLLM`：

- 后端变化会引入新的不确定变量
- 很难判断后续差异究竟来自模型本身，还是来自 backend 行为差异

因此，当前更合理的定位是：

- `vLLM` 继续作为独立验证对象
- 而不是立即替换现有主评测链路

## 7. 当前阶段结论

如果压缩成一句话：

**`vLLM` 的性能优势已经得到验证，但当前输出一致性尚不足以支撑它直接替代 `HF` 成为主评测 backend。**

## 8. 下一步建议

当前最合理的策略是：

1. 保留 `HF` 作为正式主评测链路
2. 把 `vLLM` 继续保留为候选加速方案
3. 如果后面要继续深入，就把问题单独定义成：
   - `HF vs vLLM` 的 backend 对齐实验

而不是把它直接混进模型能力比较主线里。
