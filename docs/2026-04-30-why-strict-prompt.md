# 2026-04-30 为什么后续主线转向 Strict Prompt

这份文档记录的是 `2026-04-30` 对当前项目输入协议的路线判断：

**后续主线逐步从 `default_prompt` 转向 `strict_json_prompt`。**

这里说的“转向”不是否定早期实验，而是重新定义：

- 哪一种 prompt 更适合做研究基线
- 哪一种 prompt 更适合做正式训练与正式评测协议

## 1. 为什么会有这个问题

当前任务不是开放式闲聊，也不是普通问答，而是一个强结构化诊断任务。  
目标输出明确要求：

- 合法 JSON
- 固定字段集合
- 固定枚举值
- 非空数组字段

也就是说，这本质上是一个**协议驱动型任务**。

在这种任务里，如果 prompt 本身不明确说明协议，却要求模型完全靠微调记住协议细节，会产生一个问题：

- 模型一部分能力被浪费在“猜协议”
- 而不是“在协议内给出更好的诊断内容”

## 2. 为什么不应该全靠微调来背 schema

如果系统真正上线或正式使用，用户输入前通常已经知道：

- 需要哪些字段
- 是否允许额外解释
- 是否必须只输出 JSON

因此在真实使用里，把这些约束写进 prompt 本身才更合理。

换句话说：

- prompt 负责声明协议
- 微调负责让模型在该协议下输出更好的内容

而不是：

- prompt 很弱
- 把所有协议负担都压给 SFT 去记

从工程角度看，后者是不必要的苛刻。

## 3. `default_prompt` 和 `strict_json_prompt` 各自的角色

### 3.1 `default_prompt`

用途：

- 研究基线
- 观察模型在弱约束下的原始表现
- 看 SFT 对“任务协议内化”贡献了多少

价值：

- 它能帮助我们看清 base model 的裸能力
- 也能看清微调对结构化输出的真实帮助

所以它不应该被删除。

### 3.2 `strict_json_prompt`

用途：

- 正式训练协议
- 正式推理协议
- 正式评测协议

价值：

- 更接近真实产品使用形态
- 更符合强结构化任务的工程约束
- 让模型把更多能力花在“内容质量”上，而不是“猜格式”

## 4. 为什么现在值得把主线切到 strict

当前我们已经完成了几个阶段：

- `0.5B` 流程验证
- `3B` 作为主实验候选的确认
- `4B` 的进一步 scale up 验证
- 自动规则评测
- pairwise judge

也就是说，早期用 `default_prompt` 做基线对照的价值，已经被充分挖掘。

现在更重要的问题不再是：

- base 在弱 prompt 下到底有多差

而是：

- 在真实、明确的协议条件下，哪种模型更值得继续投入
- 下一阶段应该继续 scale up，还是转向 alignment / DPO

所以从阶段目标上看，当前更适合把主线切到 strict。

## 5. 为什么不是立刻否定 default

虽然主线应逐步切到 strict，但这不意味着 default 没价值。

更合理的做法是：

- `default_prompt` 保留为研究基线
- `strict_json_prompt` 作为正式主线

也就是说：

- default 用于回看、对照、解释实验
- strict 用于后续训练、评测和偏好数据采集

## 6. 这对后续训练意味着什么

因此后续的 `3B/4B` 配置，应该补齐 strict 版本。

原因是：

- 如果正式推理和正式评测都准备使用 strict 协议
- 那训练阶段也应该至少做一轮 strict 版验证

否则就会出现：

- 训练时遵循一种协议
- 推理和评测时又换成另一种协议

这种不对齐会让后续结论变脏。

## 7. 这对后续评测意味着什么

后续评测的重点也应该逐步转到 strict 分支上。

更具体地说：

- `default_prompt` 继续保留少量对照
- 但主线自动评测、judge、DPO 候选采集，应该优先围绕 strict 版输出

这样更符合两个目标：

1. 更接近真实产品协议
2. 更公平地评估模型在明确任务约束下的内容质量

## 8. 当前建议

当前最合理的策略是：

1. 保留已有 `default_prompt` 历史结果
2. 给 `3B` / `4B` 补齐 strict 训练配置
3. 给 `3B` / `4B` 补齐 strict 推理配置
4. 后续主线优先跑 strict 版训练与评测
5. 只把 default 作为少量对照线保留

## 9. 一句话结论

对于当前这种强结构化诊断任务：

**prompt 本身就应该承担协议声明职责，因此后续主线应逐步从 `default_prompt` 转向 `strict_json_prompt`；微调负责在该协议下提升内容质量，而不是单独承担“记住 schema”的全部负担。**
