## 2. 5060 Ti 阶段的意义

`5060 Ti` 阶段最重要的价值不是“训出最终最强模型”，而是：

- 跑通全链路
- 暴露问题
- 验证训练/评测/judge/DPO 采集流程

当前已经可以明确：

- `0.5B` 更适合作为**流程验证模型**
- 它证明了：
  - 数据可训
  - 规则评测可做
  - judge 流程可做
  - DPO 候选采集可做

但它不再是后续主实验模型。

## 3. 5090 Ti 阶段的意义

`5090 Ti` 阶段的核心价值是：

- 找到比 `0.5B` 更强、更值得投入的主实验模型
- 回答继续 scale up 是否仍然有收益

当前阶段已经得到两个很关键的结论：

### 3.1 `3B` 显著优于 `0.5B`

在自动规则评测和 pairwise judge 上，`3B` 已经明显优于 `0.5B`。

### 3.2 `4B strict lora` 又显著优于 `3B strict lora`

这说明：

- 至少到 `4B` 这一档，继续 scale up 仍然有明确收益

因此当前路线已经从：

- “验证流程能不能跑”

转成了：

- “选择后续更大模型阶段的正式主实验候选”

## 4. 为什么 `strict_json_prompt` 现在固定为正式主线

当前项目已经决定：

- `strict_json_prompt` 是后续正式协议
- `default_prompt` 不再作为正式主线，只保留为研究基线

原因非常明确：

### 4.1 这是一个强结构化任务

当前任务要求模型输出：

- 合法 JSON
- 固定字段
- 固定枚举
- 非空数组字段

这类任务本来就应该把协议写清楚，而不是完全依赖微调“背格式”。

### 4.2 strict 更接近真实产品使用形态

在真实系统中，应该明确告诉模型：

- 要哪些字段
- 不要额外解释
- 只能输出 JSON

所以 strict 是更合理的正式输入协议。

### 4.3 default 的价值已经被榨干

`default_prompt` 在当前阶段仍有价值，但它的定位已经变成：

- 研究基线
- 观察 base 的裸能力
- 解释 SFT 对协议对齐的贡献

而不是：

- 后续正式训练/正式评测主线

## 5. 为什么现在不急着直接进入 DPO

当前不急着正式进入 `DPO`，原因有三层：

1. `0.5B` 已经明确不适合作为正式偏好优化主力模型
2. `3B/4B` 的 scale up 收益还没有被完全榨干
3. 当前最重要的问题仍然是：
   - 更大的模型是否还值得继续上
   - strict 主线下的内容质量上限在哪里

所以当前更合理的顺序仍然是：

1. `3B/4B` 的 strict SFT
2. 自动评测
3. pairwise judge
4. 再决定是否进入正式 `DPO`

## 6. 为什么还没有把 vLLM 接成主链路

`vLLM` 当前的定位已经比较明确：

- 值得保留
- 值得继续研究
- 但还不能直接替换 `HF` 主评测链路

原因是：

- `vLLM` 在 `4B strict lora` 上表现出明显速度优势
- 但当前小样本观察下，和 `HF` 的输出一致性仍存在不可忽略偏差

所以当前结论是：

- `vLLM` 是候选加速 backend
- 但现阶段仍以 `HF` 为正式主链路

## 7. 当前最重要的路线判断

截至现在，项目状态已经很清楚：

### 已经确定的

- `0.5B` 是流程验证模型
- `strict_json_prompt` 是正式协议主线
- `3B` 比 `0.5B` 更强
- `4B strict lora` 比 `3B strict lora` 更强

### 还没有最终确定的

- 是否继续 scale up 到更大模型
- 什么时候切到 `4090 + QLoRA`
- 什么时候正式进入 `DPO`

## 8. 下一步：迁移到 4090

当前我认为，项目已经到达一个很清楚的切换点：

- 5060 Ti 阶段的流程验证已经完成
